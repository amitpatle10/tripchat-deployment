# Kafka Design — TripChat Messaging

## What Kafka Does Here

Kafka is **not** what delivers messages to users. That's STOMP (WebSocket).

Kafka's job is to reliably get every message into the database — for offline users, message history, and future services (notifications, analytics).

---

## The Two Paths

### Hot Path — What the user waits for (~10ms)

```
Client sends message
    → Server writes to outbox table (status = PENDING)
    → Server broadcasts via STOMP to all online group members
    → Sender gets ack ✓   ← user is done waiting here
```

### Cold Path — Happens in background, user doesn't wait

```
Outbox relay wakes up (triggered by PostgreSQL LISTEN/NOTIFY)
    → Reads PENDING records from outbox table
    → Publishes to Kafka (acks=all)
    → Kafka consumer reads and persists to messages table
    → Outbox record marked PUBLISHED
```

---

## Who Delivers to Whom

```
Sender ──WebSocket──► Server ──STOMP──► Online receivers   (real-time)
                         │
                      Outbox
                         │
                       Relay ──► Kafka ──► Consumer ──► DB ──► Offline receivers
                                                               (fetch history later)
```

- **STOMP** = pushes to users online right now
- **Kafka + DB** = serves users who are offline or fetch history later

## Kafka Properties

| Property      | Value               | Why                                                                       |
| ------------- | ------------------- | ------------------------------------------------------------------------- |
| Topic         | `chat.messages`     | One topic for all group messages                                          |
| DLQ           | `chat.messages.dlq` | Failed messages go here instead of blocking the partition                 |
| Partition key | `groupId`           | All messages in same group → same partition → order guaranteed            |
| Partitions    | 3                   | Supports up to 3 parallel app servers; can increase later, never decrease |
| Retention     | 7 days              | Messages older than 7 days dropped from Kafka (already in PostgreSQL)     |
| Producer acks | `all`               | All replicas confirm before ack — no data loss. Free since relay is async |
| Delivery      | At-least-once       | Simpler than exactly-once; duplicates handled by `client_id` dedup        |
| Offset commit | Manual              | Commit only after DB write succeeds — never lose a message                |
| Retries       | 3                   | Retry on transient broker failures before sending to DLQ                  |

## Outbox Relay — How It Works

The relay is a background thread inside our Spring Boot app. Kafka doesn't poll the DB — our relay does, then pushes to Kafka.

```
PostgreSQL NOTIFY fires on every outbox INSERT
    → Relay wakes up immediately (no polling waste)
    → SELECT * FROM outbox WHERE status = PENDING LIMIT 100
    → KafkaProducer.send(message) with acks=all
    → On success: UPDATE outbox SET status = PUBLISHED
    → Safety net: slow poll every 30s catches anything missed on relay restart
```

---

## Idempotency — Handling Duplicates

If relay crashes after publishing to Kafka but before marking PUBLISHED, the same message gets published again on restart. Handled at the consumer:

```sql
INSERT INTO messages (...) VALUES (...)
ON CONFLICT (client_id) DO NOTHING
```

`client_id` is a UUID generated by the client. Two layers of dedup:

- Outbox table: unique constraint on `client_id` (no duplicate rows)
- Messages table: unique constraint on `client_id` (no duplicate persists)

## Partitions and Scaling

```
1 server  →  handles Partition 0 + 1 + 2  (all 3)
2 servers →  Server 1: P0 + P1  |  Server 2: P2
3 servers →  Server 1: P0  |  Server 2: P1  |  Server 3: P2
4 servers →  3 servers active  |  4th server idle (no partition left)
```

Max parallel consumers = number of partitions. Extra consumers beyond 3 get no work.
